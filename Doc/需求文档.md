# NewLife.MQTT 需求

## 1. 背景与目标

### 1.1 背景

物联网（IoT）行业快速发展，MQTT 协议已成为设备通信的事实标准。然而现有 .NET 生态中缺乏一款**协议完整、零依赖、客户端与服务端一体、企业功能免费**的 MQTT 实现：

- **MQTTnet**：客户端功能完善，但服务端仅为轻量级嵌入式 Broker，缺少集群、桥接、规则引擎等企业功能；断线重连后需用户手动重新订阅
- **Eclipse Paho C#**：仅客户端，依赖较多，不支持 .NET Framework 4.5 和 .NET 10
- **EMQX / Mosquitto / HiveMQ**：成熟的服务端产品，但非 .NET 技术栈，企业功能需付费或独立部署

在信创合规和国产化替代的大趋势下，亟需一款纯国产自主研发、MIT 开源、零第三方依赖的 MQTT 完整协议实现。NewLife.MQTT 由**国内新生命团队**从零自主研发，MIT 开源，所有依赖（仅 NewLife.Core）同为国产自主。

### 1.2 目标

| 目标 | 可衡量标准 |
|------|----------|
| 协议完整 | MQTT 3.1.1 全部 14 种报文实现；MQTT 5.0 全部 30 种属性 + AUTH 报文 + 所有行为语义实现 |
| 零依赖 | 仅依赖 NewLife.Core 基础库，NuGet 依赖链中无任何第三方包 |
| 客户端+服务端一体 | 单一 `NewLife.MQTT` NuGet 包同时提供完整客户端和服务端 |
| 高性能 | 单机支撑百万级连接，千万级消息吞吐 |
| 六种传输 | TCP / TLS / WebSocket / WSS / QUIC / 可靠 UDP 全覆盖 |
| 企业就绪 | 集群 / 桥接 / 规则引擎 / WebHook / ACL / $SYS 全部内置免费 |
| 全版本覆盖 | 支持 .NET Framework 4.5 ~ .NET 10 全部主流运行时 |
| 信创合规 | 纯国产自主研发 + MIT 开源 + 零第三方依赖 + 国产 OS/CPU 全支持，满足等保/信创选型要求 |

## 3. 功能需求

### 3.1 协议编解码模块

- **描述**：完整实现 MQTT 3.1.1 和 MQTT 5.0 协议的报文编解码
- **用户故事**：作为 IoT 开发者，我希望库能正确编解码所有 MQTT 报文类型，以便与任何标准 Broker 互操作
- **验收条件（AC）**：
  - [x] 14 种 MQTT 3.1.1 报文类型全部实现（CONNECT / CONNACK / PUBLISH / PUBACK / PUBREC / PUBREL / PUBCOMP / SUBSCRIBE / SUBACK / UNSUBSCRIBE / UNSUBACK / PINGREQ / PINGRESP / DISCONNECT）
  - [x] AUTH 报文（MQTT 5.0 新增，类型值 15）实现
  - [x] 固定头部（4 位类型 + 4 位标志 + 1~4 字节变长剩余长度）正确解析，支持最大 256MB 报文
  - [x] UTF-8 字符串（2 字节长度前缀）读写
  - [x] 二进制数据（2 字节长度前缀）读写
  - [x] 报文标识符（2 字节 UInt16）线程安全分配（`Interlocked.Increment`）
  - [x] 粘包拆包正确处理（`MqttCodec` + `PacketCodec` 流式解析）
  - [x] 30 种 MQTT 5.0 属性标识符全部编解码（`MqttProperties` + `MqttPropertyId` 枚举）
  - [x] 7 种 MQTT 5.0 属性数据类型全部支持（Byte / UInt16 / UInt32 / 变长整数 / UTF-8 字符串 / 二进制数据 / UTF-8 字符串对）
  - [x] 各报文类型固定标志位独立校验（`GetFlag()` 方法）
- **优先级**：Must

### 3.2 客户端模块

- **描述**：全功能 MQTT 客户端，支持多种传输协议、自动重连和自动重订阅
- **用户故事**：作为 IoT 设备开发者，我希望使用简单的 API 连接 Broker 并收发消息，以便快速完成设备接入
- **验收条件（AC）**：
  - [x] TCP 标准传输（`tcp://host:port`，默认端口 1883）
  - [x] TLS/SSL 加密传输（X509 证书，支持 pfx/pem，TLS 1.2+，默认端口 1883）
  - [x] WebSocket 传输（`ws://host:port`，默认端口 1883，`WebSocketClientCodec` 编解码）
  - [x] WSS 传输（`wss://host:port`，自动启用 TLS，默认端口 1883）
  - [x] MQTT over QUIC 传输（`quic://host:port`，.NET 7+，ALPN "mqtt"，默认端口 14567）
  - [x] MQTT over 可靠 UDP 传输（自研可靠传输层，全平台，帧格式 Data/Ack/Connect/ConnAck/Close）
  - [x] MQTT 3.1 / 3.1.1 / 5.0 版本选择（`Version` 属性，默认 V311）
  - [x] 发布 QoS 0 / 1 / 2 完整确认流程（QoS 2 四步握手：PUBLISH → PUBREC → PUBREL → PUBCOMP）
  - [x] 订阅/退订（回调绑定，按主题通配符模糊匹配分发）
  - [x] 遗嘱消息设置（`WillTopic` / `WillMessage` / `WillQoS` / `WillRetain`）
  - [x] 断线重连（指数退避 `2^n`，可配 `InitialReconnectDelay` / `MaxReconnectDelay` / `MaxReconnectAttempts`）
  - [x] 重连后自动重订阅（维护 `_subs` 字典，`ConnectAsync()` 中自动重发）
  - [x] 心跳保活（KeepAlive 半周期定时 Ping，默认 600 秒，`TimerX`）
  - [x] Inflight 消息队列 + 超时重发（`InflightManager`，DUP=1，默认 10s 超时 / 最大 3 次重发）
  - [x] 连接字符串初始化（`Init("Server=;UserName=;Password=;ClientId=;Timeout=")`）
  - [x] 连接/断开事件通知（`Connected` / `Disconnected` 事件）
  - [x] 编码器可替换（`IPacketEncoder`，默认 JSON）
  - [x] 性能跟踪（`ITracer` Span 埋点，每个 Send/Receive 操作均有 Span）
  - [x] 异步锁连接保护（`SemaphoreSlim` 双重检查，防止并发连接）
  - [x] 超时销毁重建连接（连续 3 次 `TaskCanceledException` 自动销毁并下次请求重建）
  - [x] 重复订阅防护（`_subs` 字典去重）
  - [x] 收到消息事件（`Received` 全局事件 + 订阅回调按主题通配符匹配）
- **优先级**：Must

### 3.3 服务端模块

- **描述**：高性能 MQTT Broker，支持会话管理、消息路由、安全认证和企业扩展
- **用户故事**：作为平台运维工程师，我希望部署一个高性能的 MQTT Broker，以便承载大规模设备连接
- **验收条件（AC）**：
  - [x] 会话创建与注册（`MqttExchange.Add()` ConcurrentDictionary 管理）
  - [x] 会话超时清理（30 秒定时扫描 `RemoveNotAlive()`，清除不活跃会话并断开连接）
  - [x] 持久会话（`PersistentSession` 保存订阅关系 ConcurrentDictionary + 离线消息 ConcurrentQueue，1000 条上限防溢出）
  - [x] CleanSession=0 会话恢复（`RestorePersistentSession()` 恢复订阅 + 推送离线消息）
  - [x] CleanSession=1 清除旧会话（`ClearPersistentSession()`）
  - [x] ClientId 映射查询（`RegisterClientId()` + `GetClientIds()`）
  - [x] 主题精确匹配（ConcurrentDictionary 直接查找）
  - [x] 通配符匹配（`MqttTopicFilter.IsMatch()` 按 `/` 分割逐层匹配，`+` 单层 / `#` 多层）
  - [x] `$SYS/` 系统主题屏蔽通配符订阅
  - [x] 主题过滤器验证（`IsValidTopicFilter()` 校验位置合法性）+ 发布主题名验证（`IsValidTopicName()` 禁止通配符）
  - [x] 主题最大长度 65536 字节验证
  - [x] QoS 降级分发（取发布 QoS 和订阅 QoS 较小值）
  - [x] Retain 保留消息（存储 / 清除（空 Payload TryRemove）/ 新订阅推送）
  - [x] 遗嘱消息（异常断开时通过 Exchange + ClusterExchange 发布，正常 DISCONNECT 时清除）
  - [x] 共享订阅负载均衡（`$share/{group}/{topic}` 轮询分发，`_sharedGroupIndex`）
  - [x] $SYS 系统主题（14 种标准主题定时发布，可配间隔默认 60 秒，`EnableSysTopic` 可关闭）
  - [x] 可插拔认证接口（`IMqttAuthenticator.Authenticate()`，默认 `DefaultMqttAuthenticator` 全放行）
  - [x] ACL 主题级权限控制（`AuthorizePublish()` 失败 ReasonCode=0x87 / `AuthorizeSubscribe()` 失败 QoS=0x80）
  - [x] TLS/SSL 加密（X509 证书，pfx/pem）
  - [x] DI 可扩展架构（`IMqttHandler` Transient / `IMqttExchange` Singleton / `IMqttAuthenticator` 自定义）
  - [x] 编解码器链（ProxyCodec → WebSocketServerCodec → HttpCodec → MqttCodec）
  - [ ] 磁盘级持久化（可通过 `IMqttExchange` 接口扩展 Redis/SQLite）
  - [ ] HTTP 管理 API（RESTful 查询连接/主题/统计/配置）
  - [ ] Dashboard（Web 可视化管理界面）
- **优先级**：Must（核心）/ Could（磁盘持久化、HTTP API、Dashboard）

- **用户故事**：作为业务集成商，我希望使用 MQTT 5.0 的高级特性（主题别名、流控、共享订阅、增强认证），以便优化通信效率和安全性
- **验收条件（AC）**：

  - [x] 会话过期间隔（`SessionExpiryInterval` 属性读写 + `MqttSessionCapabilities` 处理）
  - [x] 服务端断开报文（`DisconnectMessage` 支持 ReasonCode + Properties，服务端可主动断开并携带原因）
  - [x] 请求/响应模式（`ResponseTopic` + `CorrelationData` 属性完整透传）
  - [x] 共享订阅（`$share/{group}/{topic}` 解析 + `ExtractActualTopicFilter()` + 服务端轮询负载均衡）
  - [x] 订阅标识符（`SubscriptionIdentifier` 变长整数属性透传）
  - [x] 主题别名双向管理（客户端→服务端：`ResolveTopicAlias()` 解析；服务端→客户端：`AssignTopicAlias()` 自动分配）
  - [x] 流控（`ReceiveMaximum` → `CanSendQosMessage()` 检查 inflight 消息数是否超限）
  - [x] 最大报文大小（`MaximumPacketSize` 属性，`MqttSessionCapabilities` 记录）

  - [x] 服务端能力声明（`BuildConnAckProperties()` 在 CONNACK 中返回 MaximumQoS / RetainAvailable / Wildcard / SharedSubscription 等能力标志）
  - [x] 用户属性（`MqttProperties.UserProperties` 键值对列表，可多次出现）
  - [x] 服务端引用/迁移（`ServerReference` 属性支持）
  - [ ] 遗嘱延迟发布（`WillDelayInterval` 属性编解码完整，服务端延迟定时器待实现）
  - [ ] 消息过期清理（`MessageExpiryInterval` 属性编解码完整，分发时过期检查待实现）

  - [ ] SASL 认证实现（AUTH 框架完整，SCRAM-SHA-256 等具体方法待实现）
- **优先级**：Must（已完成项）/ Should（待完善项）

### 4.1 性能

|------|------|
| 连接容量 | 单机百万级长连接（已验证 400 万 TCP 长连接） |
| 消息吞吐 | 千万级 TPS（已验证 2266 万 tps） |
| 传输延迟 | TCP NoDelay 关闭 Nagle，QUIC 无队头阻塞 |

| 线程安全 | ConcurrentDictionary / Interlocked / SemaphoreSlim 无锁或轻量锁 |

## 5. 边界与约束

### 5.1 不做什么（明确排除项）

- 不实现 MQTT-SN（Sensor Networks）协议
- 不提供独立的 Web 管理前端框架（可通过 API 对接外部 UI）
- 不内置数据库持久化实现（提供 `IMqttExchange` 接口供扩展）
- 不实现 LDAP/OAuth 等外部认证协议（提供 `IMqttAuthenticator` 接口供扩展）
- 不支持 .NET Framework 4.0 及以下版本
- 服务端不接受 MQTT 3.1 连接（`ProtocolName=MQIsdp`）；收到时返回 CONNACK `0x01`（不接受的协议版本）并记录明确提示日志，引导客户端升级至 MQTT 3.1.1

### 5.2 已知限制

| 限制 | 影响 | 可扩展方案 |
|------|------|-----------|
| QUIC 传输仅客户端支持 | 服务端暂不支持 QUIC 监听 | .NET 9+ 逐步支持 QUIC 服务端 |
| 集群仅支持手动配置节点 | 扩缩容需手动修改 | 预留 Stardust / mDNS 集成接口 |
| 持久会话仅内存级 | 服务重启后丢失 | 实现 `IMqttExchange` 对接 Redis/SQLite |
| 遗嘱延迟定时器未实现 | 遗嘱立即发布 | 使用 `TimerX` 实现延迟队列 |
| 消息过期分发时未检查 | 过期消息仍推送 | 分发时检查 `MessageExpiryInterval` |

## 8. 术语表

| 术语 | 定义 |
|------|------|
| MQTT | Message Queuing Telemetry Transport，消息队列遥测传输协议，ISO/IEC 20922 标准 |
| Broker | MQTT 消息代理服务器，负责接收、路由、分发消息 |
| QoS | Quality of Service，服务质量等级（0=至多一次 / 1=至少一次 / 2=恰好一次） |
| Retain | 保留消息，Broker 为每个主题保存最后一条 Retain 消息，新订阅者立即收到 |
| Will | 遗嘱消息，客户端异常断开时由 Broker 自动发布的消息 |
| Clean Session | 清除会话标志，=1 不保留会话状态，=0 恢复持久会话 |
| Inflight | 已发送但未收到确认的 QoS>0 消息，超时后以 DUP=1 标志重发 |
| Topic Filter | 主题过滤器，支持 `+`（单层通配符）和 `#`（多层通配符） |
| Shared Subscription | 共享订阅，`$share/{group}/{topic}` 格式，同组多订阅者轮询负载均衡 |
| Topic Alias | 主题别名，MQTT 5.0 特性，用 UInt16 整数替代主题字符串以减少传输开销 |
| Receive Maximum | 接收窗口大小，MQTT 5.0 流控机制，限制 QoS>0 并发 inflight 消息数 |
| ProxyProtocol | HAProxy 提出的代理协议（v1 文本 / v2 二进制），在代理场景下透传客户端真实 IP |
| QUIC | Quick UDP Internet Connections，基于 UDP 的传输层协议，低延迟无队头阻塞 |
| DI | Dependency Injection，依赖注入，通过 `ObjectContainer` 实现可插拔架构 |
| ACL | Access Control List，访问控制列表，用于主题级别的发布/订阅权限控制 |
| $SYS | MQTT Broker 的系统主题前缀，用于发布运行统计信息（14 种标准主题） |
| SASL | Simple Authentication and Security Layer，简单认证与安全层，MQTT 5.0 增强认证支持 |
| ALPN | Application-Layer Protocol Negotiation，应用层协议协商，QUIC 传输中标识 "mqtt" |
| MoSCoW | Must / Should / Could / Won't，需求优先级分级方法 |
