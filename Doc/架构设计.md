# NewLife.MQTT 架构

## 1. 架构概览

NewLife.MQTT 采用**五层分层架构**，自底向上为：基础层、传输层、协议层、业务层、应用层。各层职责清晰、依赖单向，通过接口解耦实现可插拔扩展。

```
┌─────────────────────────────────────────────────────────────┐
│                    应用层 Application                       │
│  NewLife.MqttServer · MqttClient · AliyunMqttClient · ...  │
├─────────────────────────────────────────────────────────────┤
│                    业务层 Business                          │
│  MqttServer · MqttExchange · MqttBridge · MqttRuleEngine   │
│  MqttWebHook · ClusterServer · IMqttAuthenticator · ...    │
├─────────────────────────────────────────────────────────────┤
│                    协议层 Protocol                          │
│  MqttHandler · MqttFactory · 15种消息类型 · MqttCodec      │
│  MqttProperties(30种属性) · MqttTopicFilter · ...          │
├─────────────────────────────────────────────────────────────┤
│                    传输层 Transport                         │
│  TCP · TLS · WebSocket · WSS · QUIC · 可靠UDP · Proxy     │
├─────────────────────────────────────────────────────────────┤
│                    基础层 Foundation                        │
│       NewLife.Core (网络/序列化/日志/缓存/追踪)            │
└─────────────────────────────────────────────────────────────┘
```

| 层级 | 职责 | 核心类型 |
|------|------|---------|
| 应用层 | 面向用户的可执行程序和适配器 | `NewLife.MqttServer` 应用、`AliyunMqttClient` |
| 业务层 | 消息交换、会话管理、认证鉴权、企业功能 | `MqttServer` / `MqttExchange` / `MqttBridge` / `MqttRuleEngine` |
| 协议层 | MQTT 3.1.1/5.0 报文编解码、消息处理 | `MqttHandler` / `MqttFactory` / `MqttMessage` 系列 |
| 传输层 | 多协议传输和编解码器链 | TCP/TLS/WS/WSS/QUIC/UDP/ProxyProtocol |
| 基础层 | 网络通信、序列化、日志、追踪 | NewLife.Core、NewLife.Remoting |

## 2. 数据模型

### 2.1 消息继承体系

```
MqttMessage                     -- 基类：Type/QoS/Retain/Duplicate/Properties(5.0)
  ├── ConnectMessage             -- CONNECT(1)：ProtocolName/ProtocolLevel/ClientId/Username/Password/Will
  ├── ConnAck                    -- CONNACK(2)：ConnectReturnCode/SessionPresent/ReasonCode(5.0)
  ├── MqttIdMessage              -- 带报文标识符的中间基类（Id: UInt16）
  │   ├── PublishMessage         -- PUBLISH(3)：Topic/Payload/QoS/Retain
  │   ├── PubAck                 -- PUBACK(4)：QoS 1 确认，ReasonCode(5.0)
  │   ├── PubRec                 -- PUBREC(5)：QoS 2 接收确认
  │   ├── PubRel                 -- PUBREL(6)：QoS 2 释放（固定 QoS=1）
  │   ├── PubComp                -- PUBCOMP(7)：QoS 2 完成
  │   ├── SubscribeMessage       -- SUBSCRIBE(8)：多主题过滤器+QoS+订阅选项（固定 QoS=1）
  │   ├── SubAck                 -- SUBACK(9)：每主题授权 QoS
  │   ├── UnsubscribeMessage     -- UNSUBSCRIBE(10)：多主题退订（固定 QoS=1）
  │   └── UnsubAck               -- UNSUBACK(11)：退订确认
  ├── PingRequest                -- PINGREQ(12)：心跳请求
  ├── PingResponse               -- PINGRESP(13)：心跳响应
  ├── DisconnectMessage          -- DISCONNECT(14)：断开连接，ReasonCode+Properties（5.0 增强）
  └── AuthMessage                -- AUTH(15)：增强认证（5.0 新增），ReasonCode+AuthMethod+AuthData
```

**关键枚举类型：**

| 枚举 | 文件 | 说明 |
|------|------|------|
| `MqttType` | `Messaging/MqttType.cs` | 15 种报文类型值（1~15） |
| `QualityOfService` | `Messaging/QualityOfService.cs` | QoS 0/1/2 |
| `MqttVersion` | `Messaging/MqttVersion.cs` | V310(3) / V311(4) / V500(5) |
| `MqttPropertyId` | `Messaging/MqttPropertyId.cs` | 30 种 MQTT 5.0 属性标识符 |
| `Subscription` | `Messaging/Subscription.cs` | 订阅条目（TopicFilter + QoS + NoLocal + RetainAsPublished + RetainHandling） |

### 2.2 固定头部结构

```
┌───────────┬──────────┬─────────────────────────────────────┐
│ Byte 1    │ Byte 2-5 │ Payload                             │
├───────────┼──────────┼─────────────────────────────────────┤
│ 4bit类型  │ 剩余长度 │ 可变头部 + 有效载荷                 │
│ 4bit标志  │ (变长)   │                                     │
└───────────┴──────────┴─────────────────────────────────────┘

Byte 1 标志位：
  7 6 5 4 | 3       | 2 1     | 0
  --------+---------+---------+--------
  Type    | DUP     | QoS     | Retain
  (类型)   (重复)    (服务质量) (保留)
```

**变长编码（Remaining Length）：** 采用 1~4 字节变长编码，每字节有效 7 位，最高位为延续标志。支持最大 268,435,455 字节（约 256MB）报文。由 `MqttMessage.ReadEncodedInt()` / `WriteEncodedInt()` 方法实现。

### 2.3 MQTT 5.0 属性系统

`MqttProperties` 类管理 MQTT 5.0 的 30 种属性，采用 `Dictionary<MqttPropertyId, Object>` 存储，支持 7 种数据类型。每种属性有唯一的属性 ID（`MqttPropertyId` 枚举），在报文的可变头部中以 TLV 格式编码。

```
Properties 编码格式：
┌──────────────┬─────────────────────────────────┐
│ 属性总长度   │ 属性1 │ 属性2 │ ... │ 属性N     │
│ (变长编码)   │       │       │     │           │
└──────────────┴─────────────────────────────────┘

单个属性：
┌──────────┬────────────┐
│ 属性ID   │ 属性值     │
│ (1字节)  │ (类型相关) │
└──────────┴────────────┘
```

**30 种属性完整列表：**

| 属性ID | 名称 | 数据类型 | 服务端行为 |
|--------|------|---------|-----------|
| 0x01 | PayloadFormatIndicator | Byte | 透传 |
| 0x02 | MessageExpiryInterval | UInt32 | ⚠️ 分发时未检查过期 |
| 0x03 | ContentType | String | 透传 |
| 0x08 | ResponseTopic | String | 透传（请求/响应模式） |
| 0x09 | CorrelationData | Binary | 透传（请求/响应模式） |
| 0x0B | SubscriptionIdentifier | 变长整数 | 透传 |
| 0x11 | SessionExpiryInterval | UInt32 | `MqttSessionCapabilities` 记录 |
| 0x12 | AssignedClientIdentifier | String | 可读写 |
| 0x13 | ServerKeepAlive | UInt16 | ⚠️ 服务端未主动设置 |
| 0x15 | AuthenticationMethod | String | AUTH 报文支持 |

| 0x17 | RequestProblemInformation | Byte | 可读写 |

| 0x19 | RequestResponseInformation | Byte | 可读写 |
| 0x1A | ResponseInformation | String | 可读写 |

| 0x29 | SubscriptionIdentifierAvailable | Byte | 能力声明 |
| 0x2A | SharedSubscriptionAvailable | Byte | 能力声明 |

### 2.4 编解码器链

服务端启动时按顺序注册编解码器，形成处理链：

```
网络数据 → ProxyCodec → WebSocketServerCodec → HttpCodec → MqttCodec → MqttMessage
```

| 编解码器 | 类型/路径 | 职责 |
|---------|----------|------|
| `ProxyCodec` | `ProxyProtocol/ProxyCodec.cs` | 解析 ProxyProtocol v1（文本）/ v2（二进制），透传真实客户端 IP |
| `WebSocketServerCodec` | `Http/WebSocketServerCodec.cs` | 服务端处理 WebSocket 握手升级和 WebSocket 帧 |
| `HttpCodec` | NewLife.Core 内置 | 解析 HTTP 头部，支持 WebSocket 升级请求 |
| `MqttCodec` | `MqttCodec.cs` | MQTT 报文编解码，粘包拆包处理 |

客户端编解码器链（以 WebSocket 为例）：

```
网络数据 → WebSocketClientCodec → MqttCodec → MqttMessage
```

## 3. 接口设计

### 3.1 可替换接口一览

| 接口 | 方法/签名 | 默认实现 | 生命周期 | 说明 |
|------|----------|---------|---------|------|
| `IMqttHandler` | `Process(MqttMessage)` / `Close(String)` | `MqttHandler` | Transient（每连接一实例） | 处理 MQTT 报文，可重写 `OnConnect` / `OnPublish` / `OnSubscribe` 等虚方法 |
| `IMqttExchange` | `Add(IMqttHandler)` / `Remove(String)` / `Publish(PublishMessage)` / `Subscribe(...)` | `MqttExchange` | Singleton（全局单例） | 消息交换机，路由分发、Retain 存储、持久会话管理 |
| `IMqttAuthenticator` | `Authenticate(clientId, username, password)` / `AuthorizePublish(clientId, topic)` / `AuthorizeSubscribe(clientId, topic)` | `DefaultMqttAuthenticator`（全放行） | 自定义 | 认证鉴权 + ACL 权限控制 |
| `IPacketEncoder` | `Encode(Object)` / `Decode(Packet, Type)` | `DefaultPacketEncoder`（JSON） | 自定义 | 消息体编解码器，用于客户端负载序列化 |

### 3.2 DI 注册方式

```csharp
var services = ObjectContainer.Current;

// 注入日志
services.AddSingleton<ILog>(XTrace.Log);

// 注入处理器（每个连接一个实例）
services.AddTransient<IMqttHandler, MqttHandler>();

// 注入消息交换机（全局单例）
services.AddSingleton<IMqttExchange, MqttExchange>();

// 注入自定义认证器（可选，默认全放行）
services.AddSingleton<IMqttAuthenticator, MyAuthenticator>();
```

### 3.3 自定义处理器示例

```csharp
public class MyHandler : MqttHandler
{
    private readonly ILog _log;

    public MyHandler(ILog log) => _log = log;

    protected override ConnAck OnConnect(ConnectMessage message)
    {
        _log.Info("客户端[{0}]连接 user={1} clientId={2}",
            Session.Remote.EndPoint, message.Username, message.ClientId);
        return base.OnConnect(message);
    }

    protected override MqttIdMessage OnPublish(PublishMessage message)
    {
        _log.Info("收到消息[{0}]: {1}",
            message.Topic, message.Payload.ToStr());
        return base.OnPublish(message);
    }
}

// 注册自定义处理器
services.AddTransient<IMqttHandler, MyHandler>();
```

### 3.4 集群通信协议

集群端口（默认 2883）独立于 MQTT 端口（1883），基于 `ApiServer` / `ApiClient` 实现 RPC 通信。`ClusterController` 处理集群 RPC 请求。

| 操作 | 触发时机 | 请求方 | 数据 |
|------|---------|--------|------|
| **Join** | 节点启动 | 新节点 → 所有配置节点 | `NodeInfo`（地址/端口/集群信息） |
| **Ping** | 15 秒间隔 | 每个节点 → 所有已知节点 | 心跳保活，更新 `LastActive` |
| **Lease** | 节点停止 | 离开节点 → 所有已知节点 | 通知离开 |
| **Subscribe** | 客户端订阅 | 当前节点 → 所有节点 | `SubscriptionInfo`（主题过滤器 + QoS） |
| **Unsubscribe** | 客户端退订 | 当前节点 → 所有节点 | 主题过滤器列表 |
| **Publish** | 消息发布 | 当前节点 → 匹配节点 | `PublishInfo`（主题 + Payload + QoS） |

| 领域 | 选型 | 理由 |
|------|------|------|
| 基础框架 | NewLife.Core | 自有基础库，零外部依赖，提供网络、序列化、日志、缓存、追踪 |

| 原子操作 | Interlocked | 报文标识符分配、统计计数，无锁原子操作 |
| 异步锁 | SemaphoreSlim | 客户端连接保护，轻量异步锁（双重检查） |
| 定时器 | TimerX | NewLife.Core 高精度定时器，避免 ThreadPool 竞争 |
| 序列化 | NewLife.Core 内置 JSON / Binary | 不引入第三方 JSON 库 |
| 日志追踪 | ILog + ITracer | NewLife.Core 统一日志和 APM 追踪接口 |
| 证书处理 | System.Security.Cryptography.X509Certificates | .NET 标准 X509 证书，支持 pfx/pem |
| QUIC 传输 | System.Net.Quic | .NET 7+ 标准库，ALPN "mqtt"，.NET 9+ 稳定 |
| 主题匹配 | Trie / PruningRadixTrie | 高效前缀树，加速通配符主题匹配 |
| 开源协议 | MIT | 最宽松协议，无 Copyleft / 专利条款，信创合规 |

## 5. 关键设计决策

### 5.1 客户端+服务端一体交付

| 决策点 | 方案 | 备选方案 | 选择理由 |
|--------|------|---------|---------|
| 交付形态 | 单一 NuGet 包包含客户端+服务端 | 客户端和服务端分包 | 极简集成，用户无需判断装哪个包；`Test` 项目启动即可同时跑 Server+Client |

### 5.2 零第三方依赖

| 决策点 | 方案 | 备选方案 | 选择理由 |
|--------|------|---------|---------|
| 依赖策略 | 仅依赖 NewLife.Core | 引入 System.Text.Json 等第三方包 | 消除供应链安全风险，无许可证冲突，net45 ~ net10 全兼容 |

### 5.3 编解码器链模式

| 决策点 | 方案 | 备选方案 | 选择理由 |
|--------|------|---------|---------|
| 多协议支持 | 编解码器链（ProxyCodec → WS → HTTP → MQTT） | 每种协议独立监听端口 | 同一端口支持多协议自动识别，减少端口占用；链式可插拔 |

### 5.4 DI 可扩展架构

| 决策点 | 方案 | 备选方案 | 选择理由 |
|--------|------|---------|---------|
| 扩展机制 | 接口 + ObjectContainer DI 注入 | 事件委托 / 插件文件加载 | DI 是 .NET 标准模式；Handler 按连接 Transient、Exchange 全局 Singleton，生命周期清晰 |

### 5.5 MQTT over 可靠 UDP（独创）

| 决策点 | 方案 | 备选方案 | 选择理由 |
|--------|------|---------|---------|
| 全平台低延迟传输 | 自研可靠 UDP（4 字节帧头 + 序列号 + Ack + 超时重发） | 依赖 msquic | msquic 不支持所有平台（如 ARM Linux 4.x），自研方案全平台可用，无外部依赖 |

### 5.6 集群通信独立端口

| 决策点 | 方案 | 备选方案 | 选择理由 |
|--------|------|---------|---------|
| 集群通信 | 独立端口（2883）+ ApiServer RPC | 复用 MQTT 端口 | 集群通信与客户端连接完全隔离，互不干扰；RPC 语义比 MQTT 更适合节点间操作调用 |

### 5.7 主题匹配数据结构

| 决策点 | 方案 | 备选方案 | 选择理由 |
|--------|------|---------|---------|
| 高效主题匹配 | PruningRadixTrie + MqttTopicFilter | 线性遍历所有订阅 | 前缀树剪枝大幅减少匹配次数，适合大规模主题场景；MqttTopicFilter 处理通配符语义 |

## 6. 核心交互流程

### 6.1 客户端连接流程（CONNECT / CONNACK）

```
┌──────────┐                                    ┌──────────┐
│  Client  │                                    │  Server  │
└────┬─────┘                                    └────┬─────┘
     │                                               │
     │  1. TCP 三次握手建立连接                       │
     │───────────────────────────────────────────────>│
     │                                               │
     │  2. CONNECT 报文                               │
     │    - ProtocolName("MQTT") + ProtocolLevel      │
     │    - ClientId, Username, Password             │
     │    - KeepAlive, CleanSession                  │
     │    - Will Topic/Message/QoS/Retain            │
     │    - Properties(5.0): SessionExpiry/AuthMethod │
     │───────────────────────────────────────────────>│
     │                                               │
     │                       3. 服务端处理            │
     │                          - 检查 ProtocolName:    │
     │                            MQIsdp→CONNACK 0x01  │
     │                            拒绝并记录提示日志    │
     │                          - IMqttAuthenticator  │
     │                            .Authenticate()     │
     │                          - 创建/恢复会话       │
     │                          - 注册到 Exchange     │
     │                          - 初始化 Capabilities │
     │                                               │
     │  4. CONNACK 报文                               │
     │    - ConnectReturnCode (0=Success)             │
     │    - SessionPresent                           │
     │    - Properties(5.0): 服务端能力声明           │
     │<───────────────────────────────────────────────│
     │                                               │
     │  5. 若 CleanSession=0 且 SessionPresent=1:     │
     │     恢复订阅关系 + 推送离线消息                │
     │<───────────────────────────────────────────────│
```

### 6.2 QoS 消息交互

```
QoS 0 — 至多一次（Fire and Forget）：
  Client ──PUBLISH(QoS=0)──> Server     消息可能丢失

QoS 1 — 至少一次（带 ACK）：
  Client ──PUBLISH(QoS=1, Id=N)──> Server
  Client <──PUBACK(Id=N)────────── Server   消息至少到达一次，可能重复

QoS 2 — 恰好一次（四步确认）：
  Client ──PUBLISH(QoS=2, Id=N)──> Server   Server 存储
  Client <──PUBREC(Id=N)────────── Server   收到确认
  Client ──PUBREL(Id=N)──────────> Server   释放
  Client <──PUBCOMP(Id=N)──────── Server   完成，不丢不重
```

### 6.3 消息路由与分发

`MqttExchange.Publish()` 是消息分发的核心方法，处理流程：

1. **Retain 处理**：带 Retain 标志 → 空 Payload 清除保留消息（`TryRemove`），非空存储
2. **$SYS 独立分发**：`$SYS/` 前缀消息仅发送给显式订阅 `$SYS/` 的订阅者
3. **订阅者匹配**：遍历所有会话的主题过滤器，`MqttTopicFilter.IsMatch()` 逐层匹配
4. **共享订阅**：`$share/{group}/{topic}` 格式 → 同组轮询分发（`_sharedGroupIndex` 取模）
5. **NoLocal 过滤**（5.0）：通过 `publisherSessionId` 参数过滤，不转发给发布者自身
6. **QoS 降级**：取发布 QoS 和订阅 QoS 的较小值
7. **RetainAsPublished**（5.0）：控制是否保留原始 Retain 标志转发
8. **主题别名**（5.0）：`AssignTopicAlias()` 为高频主题自动分配别名以减少传输开销
9. **流控检查**（5.0）：`CanSendQosMessage()` 检查 inflight 数是否超过 ReceiveMaximum

### 6.4 集群消息流

```
Client A ──PUBLISH──> Node 1 ──Exchange.Publish()──> 本地订阅者
                              └─ClusterExchange.Publish()──> Node 2 ──> 本地订阅者
                                                          └> Node 3 ──> 本地订阅者
```

### 6.5 集群拓扑

```
┌─────────────┐    Join/Ping/Lease    ┌─────────────┐
│  Node A     │<─────────────────────>│  Node B     │
│  MQTT:1883  │    (ApiServer:2883)   │  MQTT:1883  │
│  Cluster:   │                       │  Cluster:   │
│    2883     │                       │    2883     │
└──────┬──────┘                       └──────┬──────┘
       │                                     │
       │         ┌─────────────┐             │
       └────────>│  Node C     │<────────────┘
                 │  MQTT:1883  │
                 │  Cluster:   │
                 │    2883     │
                 └─────────────┘
```

## 7. 传输协议架构

### 7.1 支持的传输方式

| 传输方式 | 客户端 | 服务端 | 默认端口 | 实现类 | 说明 |
|---------|--------|--------|---------|--------|------|
| TCP | ✅ | ✅ | 1883 | NetClient/NetServer | 标准 MQTT 传输 |
| TCP + TLS/SSL | ✅ | ✅ | **1883** | SslProtocol + X509 | TLS 1.2+，pfx/pem，与 TCP 共用端口 |
| WebSocket | ✅ | ✅ | **1883** | `WebSocketClientCodec` / `WebSocketServerCodec` | ws://，子协议 "mqtt"，与 TCP 共用端口 |
| WSS | ✅ | ✅ | **1883** | WS + TLS | wss:// 自动启用 TLS，与 TCP 共用端口 |
| MQTT over QUIC | ✅ | — | 14567 | `MqttQuicTransport` | .NET 7+ 平台，ALPN "mqtt" |
| MQTT over 可靠 UDP | ✅ | ✅ | **1883** | `MqttUdpTransport` | 全平台自研可靠传输层，与 TCP 共用端口 |
| ProxyProtocol v1/v2 | — | ✅ | — | `ProxyCodec` | nginx/HAProxy 透传真实 IP |

> 服务端基于 NewLife.Core `NetServer` 架构，只监听 **1883** 一个端口即可同时支持 TCP/UDP/IPv6 以及 WebSocket、TLS 等多种协议——由编解码器链（`ProxyCodec → WebSocketServerCodec → HttpCodec → MqttCodec`）自动识别协议类型。MQTT over QUIC 因协议栈差异独立使用 **14567** 端口。

### 7.2 MQTT over 可靠 UDP（独创设计）

自研可靠传输层，不依赖操作系统 QUIC/msquic 支持，全平台可用：

```
帧格式：
┌──────────┬──────────┬──────────────────────┐
│ 帧类型   │ 序列号   │ 数据载荷             │
│ (1字节)  │ (3字节)  │ (可变)               │
└──────────┴──────────┴──────────────────────┘

帧类型：Data(数据) / Ack(确认) / Connect(连接) / ConnAck(连接应答) / Close(关闭)
```

可靠性保障：序列号标识每帧 → 接收方回复 Ack → 发送方超时未收到 Ack 则重发。

## 8. MQTT 5.0 会话能力

MQTT 5.0 连接建立时，客户端和服务端通过 CONNECT/CONNACK 的 Properties 交换能力信息，由 `MqttSessionCapabilities` 记录和执行：

| 能力 | 属性 | 运行时影响 |
|------|------|-----------|
| TopicAliasMaximum | UInt16 | `ResolveTopicAlias()` 解析客户端别名；`AssignTopicAlias()` 为下发消息分配别名 |
| ReceiveMaximum | UInt16 | `CanSendQosMessage()` 检查 inflight 数是否超限，限制 QoS>0 并发数 |
| MaximumPacketSize | UInt32 | 发送前检查报文大小是否超限 |
| SessionExpiryInterval | UInt32 | 控制离线会话保活时长 |
| MaximumQoS | Byte | 发布 QoS 降级，不超过对端声明的最大 QoS |
| RetainAvailable | Byte | 功能开关，声明是否支持 Retain 消息 |

## 9. 企业扩展架构

### 9.1 消息桥接

`MqttBridge` 复用 `MqttClient` 实现双向消息桥接：

| 方向 | 说明 |
|------|------|
| In | 从远端 Broker 接收消息，注入本地 Exchange |
| Out | 从本地 Exchange 获取消息，转发到远端 Broker |
| Both | 双向同步 |

支持：主题前缀映射（添加/去除前缀）、QoS 上限控制、断线自动重连。

### 9.2 规则引擎

`MqttRuleEngine` 基于 `MqttTopicFilter` 匹配消息主题，执行 5 种动作：

| 动作 | 实现 | 说明 |
|------|------|------|
| Republish | 消息重发到新主题 | 内部转发，无需外部连接 |
| WebHook | HTTP POST 推送 | 异步非阻塞，失败重试 |
| Bridge | 桥接到远端 Broker | 复用 MqttBridge |
| Drop | 丢弃消息 | 黑名单过滤 |
| Custom | 自定义委托 | `Action<MqttRule, PublishMessage>` |

### 9.3 WebHook 事件推送

`MqttWebHook` 在关键事件发生时异步 HTTP POST 推送通知：

| 事件 | 触发时机 |
|------|---------|
| 连接（Connect） | 客户端连接成功 |
| 断开（Disconnect） | 客户端断开连接 |
| 发布（Publish） | 消息发布到 Broker |
| 投递（Deliver） | 消息投递给订阅者 |
| 订阅（Subscribe） | 客户端订阅主题 |
| 退订（Unsubscribe） | 客户端退订主题 |

失败自动重试（默认 3 次），异步非阻塞不影响主流程。

### 9.4 $SYS 系统主题

14 种标准系统主题，由 `MqttExchange` 定时发布（默认 60 秒间隔），可通过 `EnableSysTopic` 开关控制：

| 主题 | 说明 |
|------|------|
| `$SYS/broker/clients/connected` | 当前在线连接数 |
| `$SYS/broker/clients/total` | 累计连接总数 |
| `$SYS/broker/clients/maximum` | 最大同时在线连接数 |
| `$SYS/broker/messages/received` | 累计接收消息数 |
| `$SYS/broker/messages/sent` | 累计发送消息数 |
| `$SYS/broker/messages/received/persecond` | 每秒接收消息数 |
| `$SYS/broker/messages/sent/persecond` | 每秒发送消息数 |
| `$SYS/broker/bytes/received` | 累计接收字节数 |
| `$SYS/broker/bytes/sent` | 累计发送字节数 |
| `$SYS/broker/subscriptions/count` | 当前订阅数 |
| `$SYS/broker/topics/count` | 当前主题数 |
| `$SYS/broker/retained messages/count` | 保留消息数 |
| `$SYS/broker/uptime` | 运行时长（秒） |
| `$SYS/broker/version` | Broker 版本号 |

## 14. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 持久会话仅内存级 | 服务重启后丢失订阅和离线消息 | 提供 `IMqttExchange` 接口，可扩展 Redis/SQLite 持久化实现 |
| 集群缺少自动发现 | 扩缩容需手动修改配置 | 预留 Stardust / mDNS 集成接口（T004） |
| 遗嘱延迟定时器未实现 | 遗嘱立即发布，不遵守 `WillDelayInterval` | 使用 `TimerX` 实现延迟队列（T002） |
