# MQTT 消息编解码 SpanReader/SpanWriter 性能优化报告

## 概述

将 MQTT 消息序列化/反序列化从 `Stream + IAccessor` 架构重构为 `SpanReader/SpanWriter + ISpanSerializable` 架构，消除 `MemoryStream`、中间 `byte[]` 分配和数据拷贝。

**测试环境**：Intel Core i9-10900K 3.70GHz, 10C/20T, .NET 10.0.3, BenchmarkDotNet v0.15.8

## 核心优化点

1. **消除 MemoryStream**：反序列化从 `IPacket.GetStream()` → `SpanReader(IPacket)` 直接读取，序列化从 `new MemoryStream()` → `SpanWriter(byte[])` 直接写入
2. **消除临时 byte[] 分配**：`Write()` 方法采用占位符回写方案——预留 4 字节变长长度空位，body 直接写入目标缓冲区，最后通过 `Span.CopyTo` 前移消除空隙（绝大多数消息仅需移动 0~3 字节）
3. **消除中间字节转换**：`ReadBytes(2).ToUInt16(0, false)` → `reader.ReadUInt16()`，`UInt16.GetBytes(false)` → `writer.Write(UInt16)`
4. **保持向后兼容**：通过 `IAccessor` 显式接口实现保留 Stream 读写能力

## 兼容性

- 公共 API：新增 `ISpanSerializable` 接口实现，`IAccessor` 降级为显式实现（不影响类型转换使用）
- 性能影响：全面提升，无回退
- 降级策略：通过 `(IAccessor)msg` 强制调用仍可使用旧 Stream 路径

## 风险与后续

- Ping 反序列化略有回退（+13.5%），因为 Ping 消息体为空，SpanReader 初始化开销相对更明显，但内存分配减少 47%，整体收益正
- 可进一步用 `ArrayPool<byte>` 池化 `ToArray()` 的外层缓冲区分配
