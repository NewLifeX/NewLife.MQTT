# MQTT 消息序列化与反序列化性能测试

## 性能概览

本测试针对 NewLife.MQTT 库中 MQTT 消息的序列化（编码）与反序列化（解码）进行基准测试，**重点细化 Publish（发布）消息在不同载荷大小和是否携带 MQTT 5.0 Properties 下的性能表现**，同时保留其余关键消息类型作为基线参照。共覆盖 44 个测试用例。

**核心发现：**

- 序列化快速路径优化生效，小消息（body ≤ 127B）直接在目标缓冲区回填长度，避免临时缓冲区分配和拷贝
- MQTT 3.1.1 下 Publish 序列化从 0B → 4KB 载荷仅增长 66%（111 → 185 ns），载荷拷贝带来的增量约 0.018 ns/byte
- MQTT 5.0 Properties 是主要开销来源：相同载荷下 V5.0 序列化比 V3.1.1 慢约 2.6~2.8x，反序列化慢约 4.3~5.6x
- 载荷大小对反序列化几乎无影响（V3.1.1 全级别稳定在 75~85 ns，V5.0 稳定在 313~347 ns），因为 Payload 以引用方式获取而非拷贝
- 所有 V3.1.1 Publish 序列化统一分配 48B（`OwnerPacket`），V5.0 分配 96B（外层 + Properties 内层池化缓冲区）

## 测试方法

- **序列化**：调用 `MqttMessage.ToPacket()` 将消息对象编码为二进制数据包，返回数据包字节总数
- **反序列化**：调用 `MqttFactory.ReadMessage(IPacket, MqttVersion)` 将预序列化的二进制数据还原为消息对象
- 所有消息在 `[GlobalSetup]` 中预构建，消除构造开销对测量的干扰
- **Publish 测试矩阵**（重点）：
  - 载荷大小：0B（空通知）、34B（传感器 JSON）、256B（中等）、1KB（大负载）、4KB（超大负载）
  - QoS 等级：QoS0（无 Id 字段）和 QoS1（含 2 字节 Id）
  - 协议版本：V3.1.1（无 Properties）和 V5.0（携带 6 个典型属性 + 1 个 UserProperty）
- MQTT 5.0 Publish Properties 包含：PayloadFormatIndicator、MessageExpiryInterval、ContentType、ResponseTopic、CorrelationData、UserProperty
- 其余基线消息：Connect、ConnAck、PubAck、Subscribe、PingReq、Disconnect（V3.1.1）；Connect、Auth（V5.0）

## 对比分析

### 载荷大小对 Publish 性能的影响

#### 序列化：载荷拷贝开销线性增长

V3.1.1 下 Publish QoS0 序列化耗时随载荷递增：

```
0B → 34B → 256B → 1KB → 4KB
111 → 117 → 124 → 134 → 185 ns
```

- 0B → 34B：+6 ns（+5.4%），34 字节载荷拷贝开销极小
- 34B → 256B：+7 ns（+6.0%），中等载荷影响有限
- 256B → 1KB：+10 ns（+8.0%），Span 拷贝开销开始显现
- 1KB → 4KB：+51 ns（+38.1%），大负载拷贝成为主要增量
- **等效拷贝速率**：(4096 - 0)B / (185 - 111)ns ≈ **55.4 GB/s**，接近内存带宽极限

V5.0 序列化同样线性增长，但基础常量更高（308 ns vs 111 ns），因为 Properties 写入占据了固定开销。

#### 反序列化：载荷大小几乎无影响

V3.1.1 反序列化中：

```
0B（56 ns） → 34B（78 ns） → 256B（77 ns） → 1KB（76 ns） → 4KB（75 ns）
```

- 0B → 34B 有 22 ns 跳变，因为 0B 时 `Payload` 跳过创建（`reader.Available == 0`），34B+ 时需要创建 `ArrayPacket` 引用
- 34B 之后载荷大小无影响（75~78 ns 波动范围内），因为 Payload 使用 `reader.ReadPacket()` 直接切片引用原始 buffer，**零拷贝**

V5.0 反序列化同样不受载荷影响（313~347 ns 范围内），额外开销全部来自 Properties 解析。

### QoS 等级影响

QoS1 相比 QoS0 额外写入/读取一个 2 字节 Id 字段。实测 QoS 差异在 **±3% 以内**，属于统计噪声范围，对实际性能无显著影响。

### Properties 引入的额外开销

| 维度 | V3.1.1 (34B QoS0) | V5.0 (34B QoS0) | 差值 | 增长比 |
|------|------------------:|-----------------:|-----:|-------:|
| **序列化耗时** | 117.70 ns | 313.95 ns | +196.25 ns | 2.67x |
| **序列化内存** | 48 B | 96 B | +48 B | 2.00x |
| **反序列化耗时** | 78.58 ns | 340.34 ns | +261.76 ns | 4.33x |
| **反序列化内存** | 224 B | 1136 B | +912 B | 5.07x |

**Properties 开销分析：**
- 序列化端：+196 ns 来自 Properties 的 `Write()` 方法，它使用独立的池化缓冲区写入 6 个属性 + 1 个 UserProperty，然后将内容和长度前缀拷贝到外层 writer。额外的 48B 内存来自 Properties 内部的 `OwnerPacket`
- 反序列化端：+261 ns 和 +912B 来自 `new MqttProperties()` 及其内部 `Dictionary<MqttPropertyId, Object>` 的初始化、属性值的装箱（Byte→Object、UInt32→Object）、字符串读取、UserProperties 列表扩容等

**优化效果总结：**

小消息快速路径优化在 **body ≤ 127 字节的消息上效果显著**：
- PubAck/PingReq/Disconnect/ConnAck 等消息体极小的类型，序列化从 38~44 ns 降至 28~32 ns，**提升约 25%**
- 优化机制：当 `GetEstimatedBodySize() <= 127` 时，长度字段仅需 1 字节，直接在目标缓冲区写入消息体后回填长度，跳过了临时 `OwnerPacket` 的分配和拷贝
- 对大消息（Connect/Publish/Subscribe 等 body > 127B）的影响在误差范围内，因为它们不走快速路径

> Auth V5.0 序列化耗时增长（181→258 ns）可能与 `GetEstimatedBodySize()` 变更导致的分支切换有关，需进一步排查。

## 性能瓶颈定位

### 1. MqttProperties 反序列化内存分配过高（严重）

每次反序列化 V5.0 Publish 时新建 `MqttProperties` 对象，其内部 `Dictionary<MqttPropertyId, Object>` 和 `IList<KeyValuePair<String, String>>` 在解析每个属性时产生装箱和集合扩容。单个 Publish V5.0 反序列化分配 1136B，其中约 912B（80%）来自 Properties 解析。这是 V5.0 反序列化吞吐仅约 V3.1.1 的 23% 的主要原因。

### 2. MqttProperties 序列化双缓冲区写入（中等）

`MqttProperties.Write()` 使用独立的池化缓冲区先写全部属性数据，再写入变长长度前缀，最后将完整内容拷贝到外层 writer。这导致 V5.0 序列化额外分配 48B（内层 `OwnerPacket`）。对于 Publish 消息，Properties 序列化约占总耗时的 60%（~196 ns / 314 ns）。

### 3. 大载荷序列化的内存拷贝（低影响但线性增长）

4KB 载荷的 Publish 序列化比 0B 多出 74 ns（V3.1.1）和 74 ns（V5.0），是纯粹的 `Span.CopyTo` 开销。等效拷贝速率约 55 GB/s，接近硬件极限，优化空间有限。但对于 64KB+ 的超大载荷场景，可考虑引用传递而非拷贝。

### 4. 部分 V5.0 消息类型未实现属性序列化/反序列化（设计缺陷）

以下消息类型声明了 `Properties` 字段但未覆盖 `OnRead`/`OnWrite`：
- PubAck、PubRec、PubRel、PubComp
- SubAck（有 OnRead/OnWrite 但未处理 Properties）
- UnsubAck
- DisconnectMessage

这些消息在 V5.0 协议下无法正确读写 ReasonCode 和 Properties，属于功能缺陷。

### 5. ToPacket() 不支持传入协议版本（设计缺陷）

`ToPacket()` 始终传 `null` 作为 context，导致 `OnWrite` 中依赖 `context is MqttVersion` 的分支永远不会执行。目前依赖 Properties 是否为 null 来决定是否写入属性，但需要写入空属性长度（0）的场景无法覆盖。

## 优化建议

按优先级排序：

### P0（高优先级 - 功能正确性）

| 优化项 | 预期收益 | 说明 |
|--------|---------|------|
| 补全 PubAck/PubRec/PubRel/PubComp/SubAck/UnsubAck/DisconnectMessage 的 V5.0 OnRead/OnWrite 实现 | 修复 V5.0 协议兼容性 | 这些消息在 MQTT 5.0 中需要读写 ReasonCode（仅在剩余长度>0 时）和 Properties |
| 为 `ToPacket()` 增加 `ToPacket(Object? context)` 重载或传入版本参数 | 支持完整 V5.0 序列化 | 当前无法在序列化时写入空属性长度标记，violates MQTT 5.0 spec |

### P1（性能优化 - 预期高收益）

| 优化项 | 预期收益 | 说明 |
|--------|---------|------|
| MqttProperties 改用结构体字段存储常用属性 | 反序列化内存分配减少 50~70%（1136B → ~400B） | 将高频属性（MessageExpiryInterval、PayloadFormatIndicator 等）改为 nullable 字段，仅 UserProperty 等低频属性用 Dictionary 兜底，消除装箱开销 |
| MqttProperties.Write() 采用回填策略消除临时缓冲区 | Publish V5.0 序列化减少约 40~60 ns | 先预留变长长度位置，直接将属性写入目标 writer，最后回填属性总长度字段（类似 MqttMessage.Write 快速路径优化） |
| 考虑 MqttProperties 对象池化 | 减少 GC 压力 | 高吞吐场景下 Properties 对象频繁创建销毁，可复用 |

### P2（性能优化 - 预期中等收益）

| 优化项 | 预期收益 | 说明 |
|--------|---------|------|
| 字符串编码优化：缓存常用 Topic 的 UTF-8 字节 | 高频 Publish 序列化减少 10~20 ns | 对于不变的 Topic 字符串，可缓存 UTF-8 编码结果避免重复编码 |
| 超大载荷引用传递 | 消除 64KB+ 载荷拷贝 | 对于大 Payload，可使用 `IPacket` 引用拼装而非 Span 拷贝 |
| 排查 Auth V5.0 序列化回退 | 恢复 ~77 ns 性能 | Auth 序列化从 181 ns 增长到 258 ns，可能是 `GetEstimatedBodySize()` 返回值导致未进入快速路径 |

## 总结

| 指标 | MQTT 3.1.1 | MQTT 5.0 |
|------|----------:|----------:|
| 最快序列化 | Disconnect: 28 ns | Auth: 258 ns |
| 最快反序列化 | PingReq: 26 ns | Auth: 153 ns |
| 最慢序列化 | Pub 4KB: 185 ns | Pub 4KB: 382 ns |
| 最慢反序列化 | Connect: 129 ns | Connect: 368 ns |
| Pub 34B 序列化吞吐 | **849 万 msg/s** | **318 万 msg/s** |
| Pub 34B 反序列化吞吐 | **1272 万 msg/s** | **293 万 msg/s** |
| Pub 1KB 序列化吞吐 | **742 万 msg/s** | **301 万 msg/s** |
| Pub 1KB 反序列化吞吐 | **1309 万 msg/s** | **291 万 msg/s** |
| Pub 4KB 序列化吞吐 | **537 万 msg/s** | **261 万 msg/s** |

| 序列化内存（Publish 典型） | 48 B | 96 B |
| 反序列化内存（Publish 典型） | 224 B | 1136 B |

**关键结论：**

1. **小消息序列化优化生效**：PubAck/PingReq/Disconnect/ConnAck 等小消息序列化提速 25%（快速路径回填避免临时缓冲区）
2. **载荷大小对序列化线性影响、对反序列化无影响**：序列化随载荷增长线性增加（拷贝开销），反序列化通过引用切片实现零拷贝
3. **Properties 是 V5.0 性能的核心瓶颈**：序列化增加约 2.5~2.8x 开销，反序列化增加 4.3~5.6x 开销，建议优先优化 MqttProperties 数据结构
4. **QoS 等级对性能无显著影响**：QoS0/QoS1 差异在 ±3% 以内
